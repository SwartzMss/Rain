# 搜索结果连续行号设计

## 目标

搜索结果作为重新生成的日志文件显示自己的连续行号，不再把原文件行号显示在左侧。原文件行号继续作为隐藏来源元数据，仅用于右键打开原文件时精确定位。

## 行号语义

搜索结果每一行的可见编号使用当前结果集中的位置：

`可见行号 = 当前分页起点 + 当前页索引 + 1`

因此第一页从 1 开始；若每页 1000 行，第二页从 1001 开始。改变每页数量后仍按结果集中的绝对位置连续编号。

在当前结果中继续搜索会产生新的结果集，新结果文件从 1 重新编号，并在其分页之间保持连续。

## 来源定位

每条命中的 `bundle_hash`、`file_id` 和原始 `line_number` 保持不变。结果行继续通过 `data-source-line` 保存原始零基行号；右键“在原文件中打开”继续将该原始行号传给现有文件页签定位流程。

可见编号不写回命中数据、不传给来源定位函数，也不改变后端接口或临时结果文件内容。缺少来源元数据的结果仍正常显示连续编号，但不提供来源定位菜单。

## 组件边界

改动集中在 `SearchResultViewer` 的行号渲染表达式。`getSearchHitSource`、右键菜单、`BundleView` 来源打开逻辑、分页请求和后端响应均保持不变。

## 验证

- 测试原始行号为 17 的第一页首条结果显示 1，而不是 18。
- 测试分页起点为 1000 时，当前页首条结果显示 1001。
- 测试结果行仍保留 `data-source-line="17"`。
- 测试来源解析仍返回原始行号 17，证明右键定位不使用可见编号。
- 运行完整前端测试、TypeScript 检查和生产构建。
