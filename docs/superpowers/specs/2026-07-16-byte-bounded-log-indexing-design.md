# 按字节限制日志索引内存设计

## 目标

日志索引同时使用行数和实际内容字节数控制 chunk 与事务批次，使短行海量日志和超长单行日志都具有稳定的应用层内存边界。索引单行限制与文件分页预览限制分离，避免为搜索价值较低的超长内容建立巨大的 trigram FTS。

## 配置

删除 `RAIN_INDEXING_MAX_LINE_SIZE`，不提供兼容别名或回退逻辑。新增：

- `RAIN_INDEXING_MAX_INDEXED_LINE_SIZE`，默认 `256 KiB`：每个原始行进入索引的最大前缀长度。
- `RAIN_API_MAX_PREVIEW_LINE_SIZE`，默认 `8 MiB`：文件分页接口单行返回的最大前缀长度。

两个值必须大于零且能转换为当前平台的 `usize`。配置错误在启动时指出对应的新变量名称。`.env.example`、README 和发布包外置配置同步使用新名称。

## 索引批次

索引 chunk 使用两个上限：

- 最多 200 个非空清理后日志行。
- 目标内容大小 256 KiB。

`LogChunk` 在加入每行时按清理后 UTF-8 字符串长度与必要换行分隔符累计字节数，使用饱和加法。达到任一阈值后封闭 chunk。单个索引行本身可以超过 256 KiB chunk 目标，因此 chunk 的实际最大值是索引单行上限加少量分隔和截断标记开销。

索引事务使用两个提交上限：

- 最多处理 5000 个原始输入行。
- 待写封闭 chunk 的目标总大小 16 MiB。

实现维护 `lines_since_commit` 与 `pending_bytes`。每处理一个原始行增加前者；每封闭一个 chunk 增加后者。达到任一阈值时，先将当前非空 chunk 封闭，再刷新所有待写 chunk、提交事务并将两个计数归零。文件结束时刷新剩余内容。

空行仍计入行数提交阈值，确保大量空行不会产生无限事务。行偏移采样、总行数和原始字节扫描使用完整原文件语义，不受索引前缀截断影响。

## Chunk 内容与数据库写入

封闭后的 `LogChunk` 持有一次生成的完整 segment 内容，不再在写入 `log_segments` 和 `log_segments_fts` 时分别遍历行并重复 `join()`。普通表与 FTS 写入复用同一已生成内容引用；SQL 驱动或 SQLite 内部仍可能复制参数，因此 16 MiB 是应用层待写目标，不是进程 RSS 的硬上限。

现有每次 SQL 的 segment 数量上限保持不变。数据库结构、FTS schema、chunk 索引顺序与事务原子性不变。

## 超长行行为

索引读取最多保留原始行前 256 KiB，继续消费到真实换行以保持下一行边界、完整字节计数和行偏移正确。被截断的索引文本追加现有 `[line truncated]` 标记。

文件分页读取独立使用 8 MiB 预览上限，返回现有 `truncated: true` 与截断标记。文件下载读取原文件，不进行单行截断。

搜索命中仍携带原始文件行号。索引前缀截断不会改变行号、稀疏字节偏移或右键打开原文件的定位行为；超过索引前缀的文本不可被搜索命中，这是预期取舍。

## 异常与内存边界

- 字节累计使用 `saturating_add`，极端输入不会发生整数溢出。
- 单个索引行决定 chunk 瞬时最低上限，默认约为 256 KiB，而不是严格受 chunk 目标限制。
- 刷新时 SQL 参数、SQLite 页面缓存与 FTS5 token 构建仍会产生额外内存；目标是把 Rust 待提交内容限制在约 16 MiB 加一个 chunk，而不是承诺 RSS 等于 16 MiB。
- 任何数据库写入失败继续回滚当前事务并向上传流程返回现有错误。
